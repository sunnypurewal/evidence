# Versioning

`evidence` uses semantic versioning once public tags begin.

## 0.x Policy

Before `1.0`, the API is still settling. Minor releases may include source-breaking changes when they make the package simpler, safer, or more consistent. Breaking changes should be documented in release notes with a migration note.

Patch releases should be compatible bug fixes, documentation updates, or narrowly scoped improvements.

## 1.0 and Later

After `1.0`:

- Patch releases contain compatible fixes.
- Minor releases add compatible functionality.
- Major releases may include breaking API or CLI changes.

## Release Notes

Release notes should include:

- User-visible changes
- Breaking changes and migration steps
- New or changed system dependencies
- Verification performed before tagging
